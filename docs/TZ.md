## 1. Общие сведения

### 1.1. Назначение системы

**AgroHub Logistic** — веб-платформа, которая объединяет фермеров и покупателей сельскохозяйственной продукции и берёт на себя всю логистику: от оформления заказа до доставки с подтверждением получения.

Главная техническая задача — автоматически собирать десятки разрозненных заказов от разных покупателей у разных поставщиков и выстраивать из них оптимальный маршрут для каждого транспортного средства. При этом учитываются реальные дорожные условия, временны́е окна погрузки и выгрузки, а также грузоподъёмность каждой машины.

### 1.2. Масштаб (MVP)

Система проектируется для одного города или региона. Ориентировочные нагрузки:

- до 100 заказов в сутки;
- 5–20 транспортных средств в парке;
- одно депо как отправная точка всех рейсов.

### 1.3. Участники

| Роль | Кто это |
|------|---------|
| Фермер | Поставщик, размещает товары и подтверждает заказы |
| Покупатель | Делает заказы и отслеживает доставку |
| Логист | Планирует и корректирует рейсы |
| Водитель | Выполняет рейс, фиксирует погрузку и выгрузку |
| Администратор | Управляет системой, пользователями и справочниками |

## 2. Цели и задачи

Система решает четыре ключевые проблемы:

**Консолидация.** Вместо того чтобы каждый фермер отправлял свой отдельный транспорт, заказы нескольких поставщиков объединяются в один рейс — это снижает транспортные расходы на всю цепочку.

**Маршрутизация.** Порядок объезда точек строится автоматически с учётом пробок, временных окон и грузоподъёмности. Логист получает готовый план, а не составляет его вручную.

**Контроль факта.** Водитель фиксирует погрузку через QR-код фермера и принимает электронную подпись получателя при выгрузке. Все расхождения фиксируются в системе.

**Прозрачность.** Каждый участник видит актуальный статус своего заказа и примерное время доставки.

## 3. Роли пользователей

### 3.1. Фермер

Регистрируется самостоятельно, получает доступ после верификации администратором. Управляет своим каталогом товаров и лотов, указывает адрес отгрузки и доступное время. Подтверждает или отклоняет входящие заказы. Видит статусы своих отгрузок и рейтинг от покупателей.

### 3.2. Покупатель

Регистрируется самостоятельно, указывает адрес доставки. Просматривает каталог, формирует корзину из товаров разных фермеров, оформляет заказ с указанием желаемого окна доставки. Отслеживает статус и ETA. Оставляет отзывы на фермеров после получения заказа.

### 3.3. Логист

Создаётся администратором. Запускает автоматическое планирование рейсов или корректирует его вручную через drag-and-drop интерфейс. Наблюдает за маршрутами на карте с цветовой индикацией нарушений по ETA. Получает автоматические предупреждения при конфликтах временных окон.

### 3.4. Водитель

Создаётся администратором. Работает исключительно через мобильный PWA-интерфейс, который должен функционировать без постоянного подключения к сети. Видит список точек своего маршрута, сканирует QR-коды фермеров на погрузке, принимает подпись получателя на выгрузке.

### 3.5. Администратор

Верифицирует и блокирует пользователей. Управляет справочниками (категории, единицы измерения). Просматривает аналитику. Настраивает параметры системы (координаты депо, радиус охвата, коэффициенты VRP). Может запускать планирование рейсов вручную на любую дату.

## 4. Архитектура системы

### 4.1. Компоненты

```
┌─────────────────────────────────────────────────┐
│                    Клиенты                       │
│   Browser SPA (React)    Mobile PWA (водитель)   │
└────────────────┬────────────────┬────────────────┘
                 │ HTTPS          │ HTTPS
┌────────────────▼────────────────▼────────────────┐
│          FastAPI (Uvicorn, ASGI)                  │
│                                                   │
│  Auth  │ Products │ Orders │ Trips │ Analytics    │
│                                                   │
│        FastAPI BackgroundTasks                    │
│        (маршрутизация, уведомления)               │
│                                                   │
│        APScheduler (cron: 18:00 D-1)             │
└──────────────┬────────────────────────────────────┘
               │
┌──────────────▼────────────────────────────────────┐
│   PostgreSQL 15 + PostGIS    MinIO (S3)            │
└───────────────────────────────────────────────────┘
               │
┌──────────────▼────────────────────────────────────┐
│   Яндекс.Маршрутизация API                        │
│   SMTP / SMS (уведомления)                        │
└───────────────────────────────────────────────────┘
```

### 4.2. Сервисы

| Сервис | Технология | Назначение |
|--------|-----------|-----------|
| Web API | FastAPI + Uvicorn | Основной бэкенд, REST API |
| Фоновые задачи | FastAPI BackgroundTasks + APScheduler | Асинхронные расчёты и cron |
| База данных | PostgreSQL 15 + PostGIS | Хранение данных и гео-запросы |
| Файловое хранилище | MinIO (S3-compatible) | PDF-документы, QR-коды, изображения |

### 4.3. Асинхронная обработка без брокера

Для задач, не требующих персистентной очереди (MVP), используется следующая схема:

- **FastAPI BackgroundTasks** — для задач в рамках одного запроса (отправка уведомлений, генерация PDF после доставки).
- **APScheduler** — встроен в процесс FastAPI, запускает cron-задачу планирования рейсов в 18:00 каждого дня. Состояние задач сохраняется в PostgreSQL (JobStore).
- Задачи планирования маршрутов запускаются через APScheduler и выполняются в отдельном thread pool (не блокируют event loop).

## 5. Схема базы данных

### 5.1. Пользователи (`users`)

```sql
CREATE TABLE users (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email         VARCHAR(255) UNIQUE NOT NULL,
    phone         VARCHAR(20),
    password_hash TEXT NOT NULL,
    full_name     VARCHAR(255) NOT NULL,
    role          VARCHAR(20) NOT NULL CHECK (role IN ('farmer','customer','logistician','driver','admin')),
    status        VARCHAR(10) NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','active','blocked')),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 5.2. Местоположения (`locations`)

Хранит адреса фермеров (точки отгрузки) и покупателей (точки доставки).

```sql
CREATE TABLE locations (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id),
    address     TEXT NOT NULL,
    coordinates GEOMETRY(Point, 4326) NOT NULL,
    is_primary  BOOLEAN NOT NULL DEFAULT true,
    label       VARCHAR(100)
);

CREATE INDEX idx_locations_geo ON locations USING GIST(coordinates);
```

Каждый пользователь с ролью `farmer` или `customer` обязан иметь хотя бы одну запись в `locations` — это проверяется при верификации аккаунта.

### 5.3. Транспортные средства (`vehicles`)

```sql
CREATE TABLE vehicles (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    plate_number VARCHAR(20) UNIQUE NOT NULL,
    model        VARCHAR(100),
    capacity_kg  NUMERIC(10,2) NOT NULL,
    refrigerated BOOLEAN NOT NULL DEFAULT false,
    status       VARCHAR(15) NOT NULL DEFAULT 'available'
                 CHECK (status IN ('available','in_trip','maintenance'))
);
```

### 5.4. Товары (`products`)

```sql
CREATE TABLE products (
    id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    farmer_id         UUID NOT NULL REFERENCES users(id),
    parent_product_id UUID REFERENCES products(id),
    name              VARCHAR(255) NOT NULL,
    description       TEXT,
    category          VARCHAR(100) NOT NULL,
    unit              VARCHAR(10) NOT NULL CHECK (unit IN ('kg','ton','piece')),
    price_per_unit    NUMERIC(12,2) NOT NULL,
    available_window  TSRANGE NOT NULL,
    is_active         BOOLEAN NOT NULL DEFAULT true,
    image_url         TEXT,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

**Лоты.** Запись с `parent_product_id IS NULL` — родительский лот. Дочерние SKU ссылаются на него через `parent_product_id`. При заказе лота создаётся единая позиция.

### 5.5. Заказы (`orders`)

Бизнес-правило: **один заказ = один поставщик**. Один checkout покупателя создаёт столько заказов, сколько разных фермеров в корзине.

```sql
CREATE TABLE orders (
    id                    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    customer_id           UUID NOT NULL REFERENCES users(id),
    farmer_id             UUID NOT NULL REFERENCES users(id),
    checkout_id           UUID NOT NULL,
    desired_delivery_time TSTZRANGE NOT NULL,
    total_weight_kg       NUMERIC(10,2) NOT NULL,
    total_amount          NUMERIC(12,2) NOT NULL,
    status                VARCHAR(25) NOT NULL DEFAULT 'draft'
                          CHECK (status IN (
                              'draft','confirmed_by_farmer','ready_for_shipment',
                              'in_transit','delivered','dispute','cancelled'
                          )),
    dispute_reason        TEXT,
    created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 5.6. Позиции заказа (`order_items`)

```sql
CREATE TABLE order_items (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    order_id        UUID NOT NULL REFERENCES orders(id),
    product_id      UUID NOT NULL REFERENCES products(id),
    quantity        NUMERIC(10,3) NOT NULL,
    unit_price      NUMERIC(12,2) NOT NULL,
    actual_quantity NUMERIC(10,3)
);
```

### 5.7. Рейсы (`trips`)

```sql
CREATE TABLE trips (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    driver_id       UUID NOT NULL REFERENCES users(id),
    vehicle_id      UUID NOT NULL REFERENCES vehicles(id),
    departure_time  TIMESTAMPTZ NOT NULL,
    planned_date    DATE NOT NULL,
    status          VARCHAR(10) NOT NULL DEFAULT 'planned'
                    CHECK (status IN ('planned','active','finished','cancelled')),
    total_weight_kg NUMERIC(10,2),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 5.8. Точки маршрута (`waypoints`)

```sql
CREATE TABLE waypoints (
    id                 UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    trip_id            UUID NOT NULL REFERENCES trips(id),
    order_id           UUID NOT NULL REFERENCES orders(id),
    location_id        UUID NOT NULL REFERENCES locations(id),
    type               VARCHAR(10) NOT NULL CHECK (type IN ('pickup','dropoff')),
    sequence           INTEGER NOT NULL,
    time_window_start  TIMESTAMPTZ NOT NULL,
    time_window_end    TIMESTAMPTZ NOT NULL,
    planned_arrival    TIMESTAMPTZ,
    actual_arrival     TIMESTAMPTZ,
    UNIQUE(trip_id, sequence)
);
```

### 5.9. Рейтинги (`reviews`)

```sql
CREATE TABLE reviews (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    order_id    UUID NOT NULL REFERENCES orders(id),
    reviewer_id UUID NOT NULL REFERENCES users(id),
    target_id   UUID NOT NULL REFERENCES users(id),
    rating      SMALLINT NOT NULL CHECK (rating BETWEEN 1 AND 5),
    comment     TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE(order_id, reviewer_id)
);
```

### 5.10. Уведомления (`notifications`)

```sql
CREATE TABLE notifications (
    id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id  UUID NOT NULL REFERENCES users(id),
    type     VARCHAR(50) NOT NULL,
    payload  JSONB NOT NULL,
    is_read  BOOLEAN NOT NULL DEFAULT false,
    sent_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 5.11. Задачи планирования (`planning_jobs`)

Хранит состояние APScheduler и историю запусков планирования.

```sql
CREATE TABLE planning_jobs (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    planned_for DATE NOT NULL,
    status      VARCHAR(10) NOT NULL DEFAULT 'pending'
                CHECK (status IN ('pending','running','done','failed')),
    started_at  TIMESTAMPTZ,
    finished_at TIMESTAMPTZ,
    error_text  TEXT
);
```

## 6. Функциональные требования

### 6.1. Аутентификация и авторизация

#### Регистрация

- `POST /api/v1/auth/register` — поля: `email`, `phone`, `password`, `full_name`, `role` (только `farmer` или `customer`).
- После регистрации статус — `pending`. Доступ заблокирован до верификации администратором.
- Аккаунты водителей и логистов создаются только администратором (`POST /api/v1/admin/users`).

#### Вход и токены

- `POST /api/v1/auth/login` — возвращает `access_token` (TTL 15 мин) и `refresh_token` (TTL 30 дней).
- `refresh_token` передаётся в httpOnly-cookie. `access_token` хранится в памяти клиента.
- `POST /api/v1/auth/refresh` — выдать новую пару токенов.
- `POST /api/v1/auth/logout` — инвалидировать refresh-токен (запись в таблицу отозванных токенов в PostgreSQL).

#### Права доступа

| Действие | farmer | customer | logistician | driver | admin |
|----------|:------:|:--------:|:-----------:|:------:|:-----:|
| Управление своим каталогом | ✓ | — | — | — | ✓ |
| Просмотр каталога | ✓ | ✓ | ✓ | — | ✓ |
| Создание заказов | — | ✓ | — | — | — |
| Подтверждение заказа (фермер) | ✓ | — | — | — | ✓ |
| Планирование рейсов | — | — | ✓ | — | ✓ |
| Исполнение рейса | — | — | — | ✓ | ✓ |
| Управление пользователями | — | — | — | — | ✓ |
| Просмотр аналитики | — | — | ✓ | — | ✓ |

### 6.2. Управление каталогом (Фермер)

**Одиночный товар:**

- `POST /api/v1/products` — создать SKU.
- `PUT /api/v1/products/{id}` — обновить.
- `DELETE /api/v1/products/{id}` — мягкое удаление (`is_active = false`).
- `GET /api/v1/products?category=&farmer_id=&available_from=&available_to=&limit=&offset=` — каталог с фильтрами.

**Пакетные лоты:**

- `POST /api/v1/products/batch` — создать лот.

```json
{
  "name": "Борщевой набор",
  "pickup_location_id": "uuid",
  "available_window": ["2026-06-01T00:00:00Z", "2026-06-30T23:59:59Z"],
  "items": [
    { "product_id": "uuid", "quantity": 1.5 },
    { "product_id": "uuid", "quantity": 2.0 }
  ]
}
```

### 6.3. Корзина и оформление заказов (Покупатель)

**Корзина** хранится на клиенте (Zustand + localStorage). Сервер не хранит черновые корзины.

**Checkout:**

- `POST /api/v1/orders/checkout`

```json
{
  "delivery_location_id": "uuid",
  "desired_delivery_window": {
    "start": "2026-06-05T09:00:00Z",
    "end":   "2026-06-05T18:00:00Z"
  },
  "items_by_farmer": {
    "farmer_uuid_1": [
      { "product_id": "uuid", "quantity": 5 }
    ],
    "farmer_uuid_2": [
      { "product_id": "uuid", "quantity": 10 }
    ]
  }
}
```

Все заказы создаются в одной транзакции (один `checkout_id`). При ошибке для любого фермера — полный откат. После успеха запускается `BackgroundTask`, проверяющая попадание даты доставки в ближайший слот планирования.

**Управление статусами фермером:**

- `PATCH /api/v1/orders/{id}/confirm` — `draft → confirmed_by_farmer`.
- `PATCH /api/v1/orders/{id}/ready` — `confirmed_by_farmer → ready_for_shipment`.

При смене статуса — уведомление покупателю и логисту.

### 6.4. Управление рейсами (Логист)

- `GET /api/v1/trips?date=` — список рейсов на дату.
- `GET /api/v1/trips/unassigned?date=` — нераспределённые заказы.
- `POST /api/v1/logistics/plan` — запустить автоматическое планирование. Задача выполняется асинхронно через APScheduler/BackgroundTasks. Возвращает `job_id` для отслеживания статуса.
- `GET /api/v1/logistics/plan/{job_id}` — статус задачи планирования.
- `POST /api/v1/trips/{trip_id}/add_order` — добавить заказ в рейс вручную. При нарушении временного окна — `409 Conflict`.
- `DELETE /api/v1/trips/{trip_id}/orders/{order_id}` — убрать заказ из рейса.
- `POST /api/v1/trips/{trip_id}/recalculate` — пересчёт порядка точек. **Ответ ≤ 2 сек.**
- `GET /api/v1/trips/{trip_id}/route` — маршрут в GeoJSON.

### 6.5. Исполнение рейса (Водитель)

- `GET /api/v1/driver/trip` — текущий рейс.
- `GET /api/v1/driver/trip/waypoints` — список точек с деталями.
- `POST /api/v1/driver/waypoints/{id}/arrive` — отметить прибытие.
- `POST /api/v1/driver/waypoints/{id}/pickup` — подтвердить погрузку с фактическим количеством.
- `POST /api/v1/driver/waypoints/{id}/dropoff` — принять подпись получателя (base64).
- `POST /api/v1/driver/waypoints/{id}/dropoff_offline` — отложенная отправка при появлении сети.

- Все эндпоинты: `/api/v1/…`
- Формат данных: JSON.
- Аутентификация: `Authorization: Bearer <access_token>`.

### 7.2. Коды ответов

| Код | Когда используется |
|-----|--------------------|
| 200 | Успешный запрос |
| 201 | Ресурс создан |

| 400 | Некорректные входные данные |
| 401 | Не аутентифицирован |
| 403 | Недостаточно прав |
| 404 | Ресурс не найден |
| 409 | Конфликт бизнес-правила |
| 422 | Ошибка валидации Pydantic |
| 503 | Внешний сервис недоступен |

}

## 8. Алгоритм маршрутизации (VRP)

### 8.1. Инициация

- **Автоматически:** APScheduler запускает задачу каждый день в 18:00 (D-1) для планирования на следующий день.
- **Вручную:** `POST /api/v1/logistics/plan` с телом `{"date": "2026-06-05"}`.

Задача выполняется в thread pool, чтобы не блокировать event loop FastAPI. Прогресс сохраняется в таблицу `planning_jobs`.

### 8.2. Входной набор данных

1. Все заказы со статусом `ready_for_shipment` на указанную дату.
2. Доступные ТС (`status = 'available'`) с параметрами `capacity_kg` и `refrigerated`.
3. Координаты депо из системных настроек.

### 8.3. Шаг 1 — Геофильтрация

PostGIS-запрос: отбрать заказы, где адреса поставщика и покупателя находятся в радиусе **X км** от депо. Параметр X задаётся администратором, по умолчанию 100 км.

```sql
SELECT o.id
FROM orders o
JOIN locations pl ON pl.user_id = o.farmer_id   AND pl.is_primary = true
JOIN locations dl ON dl.user_id = o.customer_id  AND dl.is_primary = true
WHERE
    o.status = 'ready_for_shipment'
    AND ST_DWithin(pl.coordinates::geography, :depot_point::geography, :radius_m)
    AND ST_DWithin(dl.coordinates::geography, :depot_point::geography, :radius_m);
```

Заказы вне радиуса переводятся в буфер с уведомлением администратору.

### 8.4. Шаг 2 — Распределение по ТС (Bin Packing)

Алгоритм First Fit Decreasing (FFD):

- Заказы сортируются по убыванию веса.
- Каждый заказ назначается в первое ТС, где достаточно места.
- Ограничение: `∑ weight ≤ capacity_kg × 0.9` (коэффициент запаса).
- Заказы с охлаждаемыми товарами назначаются только на ТС с `refrigerated = true`.

Если суммарный объём превышает вместимость всего парка — лишние заказы остаются в буфере, администратору отправляется уведомление «Требуется привлечение наёмного транспорта».

### 8.5. Шаг 3 — Запрос к VRP-движку

**Библиотека:** `google/or-tools` (Python, локально).

**Матрица расстояний:** запрашивается через **Яндекс.Маршрутизация API (Matrix Router)** с учётом пробок на время выезда. Результат кэшируется в PostgreSQL на 1 час.

**Параметры задачи:**

```
Узлы     = [депо] + [все pickup] + [все dropoff]
Начало/конец каждого маршрута = депо
Ограничения:
  - Временны́е окна на каждый узел
  - Вместимость каждого ТС
Цель: минимизировать суммарное время всех рейсов
Штрафы:
  - Опоздание к dropoff-точке: коэффициент 10
  - Ранний приезд на pickup:   коэффициент 1
Порядок: сначала все pickup, затем dropoff (Backhaul constraint)
```

**Выходные данные:** для каждого ТС — упорядоченный список waypoints с `sequence` и `planned_arrival`. Записываются в БД, создаётся рейс со статусом `planned`. Логисту отправляется уведомление.

#### Drag-and-drop

1. Логист перетаскивает карточку заказа из «Нераспределённых» в карточку машины.
2. Фронтенд: `POST /api/v1/trips/{trip_id}/add_order`.

### 10.1. Технические требования

### 10.3. Экран «Погрузка»

1. Кнопка «Прибыл» — фиксирует `actual_arrival`.
2. Кнопка «Сканировать QR-код фермера» — открывает камеру. После успешного сканирования — разблокируется список товаров.
3. Список позиций заказа с кнопками `−` / `+` для корректировки фактического количества.
4. Кнопка «Подтвердить загрузку». При расхождении — предупреждение «Будет создан спор по заказу», заказ переходит в статус `dispute`.

### 10.4. Экран «Выгрузка»

1. Кнопка «Прибыл» — фиксирует `actual_arrival`.
2. Список доставляемых позиций с плановым количеством.
3. Поле подписи (`<canvas>`): получатель расписывается пальцем или стилусом. Кнопка «Очистить» для повтора.
4. Кнопка «Завершить доставку»:
   - Онлайн: `POST /api/v1/driver/waypoints/{id}/dropoff`.
   - Офлайн: сохранить в IndexedDB, зарегистрировать Background Sync, отправить при появлении сети.
5. После успеха — статус заказа `delivered`, экран с подтверждением.

- Задачи планирования: при сбое APScheduler делает 3 повторные попытки (интервалы 30 / 60 / 120 сек), затем помечает задачу как `failed` и уведомляет администратора.
- При недоступности Яндекс.API: ответ `503 Service Unavailable` с описанием ошибки.

### 16.3. Безопасность

- Пароли: bcrypt, `cost = 12`.
- Только HTTPS (TLS 1.2+).
- Все SQL-запросы — только через ORM или явную параметризацию (без конкатенации строк).
- Загрузка файлов: проверка MIME-типа и ограничение по размеру (изображения ≤ 5 МБ, подписи ≤ 10 МБ).
- Rate limiting: 100 запросов/мин для неавторизованных, 1000/мин для авторизованных.

### 16.4. Совместимость браузеров

- SPA: Chrome 120+, Firefox 120+, Safari 17+, Edge 120+.
- PWA (водитель): iOS Safari 16+, Android Chrome 120+.

## 17. Критические сценарии и Edge Cases

### 17.1. Расхождение при погрузке

**Ситуация:** водитель ввёл фактическое количество меньше заказанного.

**Реакция системы:**
1. `order_item.actual_quantity` обновляется.
2. Статус заказа → `dispute`, причина `underpickup`.
3. Уведомления: фермеру, покупателю, логисту.
4. Вес рейса пересчитывается.
5. Маршрут **не перестраивается**, если эта точка была последней погрузкой в рейсе.

### 17.2. Конфликт временного окна

**Ситуация:** логист вручную добавил заказ в рейс, но его временное окно несовместимо с существующими точками.

**Реакция системы:**
1. Бэкенд возвращает `409 Conflict`:
```json
{
  "error": "time_window_violation",
  "detail": {
    "reason": "time_window_violation",
    "conflicting_order": "uuid",
    "conflicting_waypoint_sequence": 3,
    "message": "Заказ не вписывается в временное окно рейса"
  }
}
```
2. Карточка заказа возвращается в «Нераспределённые».
3. Конфликтующая точка на карте подсвечивается красным.
4. Toast-уведомление с деталями конфликта.

### 17.3. Нехватка транспорта

**Ситуация:** суммарный вес заказов на день превышает вместимость всего парка.

**Реакция системы:**
1. FFD-алгоритм распределяет максимально возможный объём.
2. Лишние заказы остаются со статусом `ready_for_shipment` и флагом `queued = true`.
3. Уведомление администратору: «Требуется наёмный транспорт. В очереди: N заказов, M кг».
4. В планировщике — отдельная колонка «Очередь».

### 17.4. Офлайн-подпись водителя

**Ситуация:** сети нет в момент завершения выгрузки.

**Реакция системы:**
1. Данные подписи и время прибытия сохраняются в IndexedDB.
2. Service Worker регистрирует Background Sync.
3. При появлении сети — автоматическая отправка `POST /api/v1/driver/waypoints/{id}/dropoff_offline`.
4. Сервер применяет данные, статус заказа → `delivered`.

### 17.5. Недоступность Яндекс.API при планировании

**Ситуация:** внешний API недоступен в момент запуска расчёта маршрутов.

**Реакция системы:**
1. APScheduler делает 3 попытки: через 30, 60 и 120 секунд.
2. После исчерпания попыток: `planning_jobs.status = 'failed'`, уведомление администратору.
3. Администратор может повторно запустить планирование вручную через API или интерфейс.

## 18. Форматы данных

### 18.1. GeoJSON для гео-ответов

Все ответы, содержащие координаты, строго соответствуют GeoJSON FeatureCollection (RFC 7946):

```json
{
  "type": "FeatureCollection",
  "features": [
    {
      "type": "Feature",
      "geometry": {
        "type": "Point",
        "coordinates": [37.6, 55.7]
      },
      "properties": {
        "waypoint_id": "uuid",
        "order_id": "uuid",
        "type": "pickup",
        "sequence": 1,
        "planned_arrival": "2026-06-05T09:30:00Z",
        "time_window_start": "2026-06-05T09:00:00Z",
        "time_window_end": "2026-06-05T10:00:00Z",
        "status_color": "green"
      }
    }
  ]
}
```

`status_color`: `green` — запас более 15 минут, `yellow` — 0–15 минут, `red` — опоздание.

### 18.2. Временные диапазоны

Всё время — UTC, формат ISO 8601:

```json
{
  "desired_delivery_window": {
    "start": "2026-06-05T09:00:00Z",
    "end":   "2026-06-05T18:00:00Z"
  }
}
```

### 18.3. QR-коды для погрузки

QR-код фермера содержит JSON с HMAC-подписью:

```json
{
  "farmer_id": "uuid",
  "location_id": "uuid",
  "signature": "hmac_sha256_token"
}
```

Сервер верифицирует `signature`, чтобы исключить подделку.

## 19. Глоссарий

| Термин | Определение |
|--------|-------------|
| **SKU** | Единица номенклатуры товара |
| **Лот** | Пакетное предложение из нескольких SKU (например, «Борщевой набор») |
| **Рейс (Trip)** | Маршрут одного ТС на один день |
| **Waypoint** | Точка маршрута: погрузка (pickup) или выгрузка (dropoff) |
| **ETA** | Расчётное время прибытия в точку |
| **VRP** | Vehicle Routing Problem — задача маршрутизации транспортных средств |
| **Bin Packing** | Задача оптимального распределения грузов по ТС |
| **Депо** | Отправная и конечная точка всех рейсов |
| **Checkout** | Оформление заказа покупателем; создаёт 1..N заказов |
| **Dispute** | Статус заказа при расхождении фактического и заказанного количества |
| **PostGIS** | Расширение PostgreSQL для геопространственных данных |
| **PWA** | Progressive Web Application — веб-приложение с офлайн-режимом |
| **Background Sync** | API Service Worker для отложенной отправки данных при появлении сети |
| **APScheduler** | Python-библиотека для периодических задач (замена cron) |
| **FFD** | First Fit Decreasing — алгоритм жадной упаковки |
