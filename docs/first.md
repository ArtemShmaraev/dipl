Техническое задание для разработки: «AgroHub Logistic»
1. Абстрактное описание системы
Необходимо разработать многоуровневое веб-приложение, представляющее собой маркетплейс сельскохозяйственной продукции с автоматическим модулем консолидации грузов и построения маршрутов (VRP).
Ключевой технологический вызов: Автоматическое преобразование множества разрозненных заказов от разных покупателей у разных продавцов в единый рейс транспортного средства с учетом топологии местности, временных окон погрузки/разгрузки и вместимости ТС.
2. Архитектура базы данных (PostgreSQL + PostGIS)
2.1. Пользователи (users)
Стандартная модель с наследованием ролей через ForeignKey на профиль.
•	id, role (Enum: farmer, customer, logistician, driver, admin).
•	Обязательная верификация: поле status (Enum: pending, active, blocked).
•	Особенность для farmer и customer: Наличие связи с таблицей locations.
o	locations должна хранить географические координаты в поле типа GEOMETRY(Point, 4326) для пространственных индексов.
2.2. SKU и Офферы (products)
•	id, farmer_id.
•	Параметрические характеристики: category, unit (кг/т/шт).
•	Поле available_window: tsrange (диапазон доступности товара во времени).
2.3. Корзина и Заказы (Критическая бизнес-логика)
•	Модель Заказа (orders). Должна реализовывать логику «Один заказ = Один поставщик», но «Один документ покупателя = Несколько заказов».
o	id, customer_id, farmer_id.
o	desired_delivery_time: tstzrange (желаемое окно доставки). Важно для алгоритма маршрутизации.
o	status: Enum (draft, confirmed_by_farmer, ready_for_shipment, in_transit, delivered).
2.4. Рейсы и Маршруты (Ядро модуля)
•	Рейс (trips): id, driver_id, vehicle_id, departure_time, status (planned, active, finished).
•	Точки маршрута (waypoints):
o	trip_id, order_id, location_id.
o	type: Enum (pickup - загрузка, dropoff - выгрузка).
o	sequence: Integer (порядковый номер объезда).
o	planned_arrival: Timestamp (Расчетное время).
o	actual_arrival: Timestamp (Фактическое, вводится водителем).
3. Функциональная спецификация API (Backend)
3.1. Управление каталогом (Фермер)
•	Endpoint: POST /api/v1/products/batch
•	Логика: Фермер может создавать пакетные предложения (Лоты). Например, «Борщевой набор». Это не просто товар, это группа SKU, продаваемых одной позицией с одним адресом отгрузки. Реализовать как parent_product_id внутри таблицы products.
3.2. Процесс создания заявки (Покупатель)
•	Endpoint: POST /api/v1/orders/checkout
•	Входные данные: JSON объект, где ключ — farmer_id, значение — массив выбранных SKU.
•	Транзакционное требование: Атомарно создать N заказов в статусе confirmed.
•	Триггер: После успешного создания, система должна проверить, не входит ли дата доставки в ближайший слот планирования рейсов (например, «завтра»).
3.3. Алгоритм построения маршрутов (Сервис Маршрутизации)
Это отдельный микросервис или асинхронный воркер (Celery).
•	Инициация: POST /api/v1/logistics/plan (Trigger by Admin/Operator or Cron at 18:00 D-1).
•	Входной набор данных (DataSet):
1.	Все заказы со статусом ready_for_shipment на конкретную дату.
2.	Парк машин (Vehicle Fleet) с параметрами capacity_kg, refrigerated.
•	Шаг 1: Фильтрация по гео-хэшу.
o	Запрос в PostGIS: выбрать заказы, где поставщики и заказчики находятся в радиусе X км от депо или друг от друга (кластеризация).
•	Шаг 2: Решение задачи упаковки (Bin Packing).
o	Сопоставление заказов с конкретными ТС, чтобы суммарный вес заказов ≤ capacity_kg * 0.9.
•	Шаг 3: Запрос к VRP-движку.
o	Формирование запроса к библиотеке google/or-tools или VROOM.
o	Параметры:
	Матрица «Время-Расстояние» (запросить через API карт).
	Штраф за опоздание к dropoff-точкам должен быть в 10 раз выше, чем штраф за ранний приезд на pickup.
	Настройка: Сначала обязательный объезд всех pickup в цепочке, затем dropoff (Backhauls).
•	Выходные данные: Упорядоченный список waypoints для каждой машины с указанием времени ETA.
4. Интерфейс пользователя (Frontend SPA)
4.1. Компонент «Планировщик рейсов» (Роль Логист)
•	Визуализация: React-Kanban или React-Flow доска.
•	Слева — «Пулы нераспределенных заказов» (карточки заказов).
•	Справа — «Карточки машин».
•	Механика Drag-and-Drop:
o	Логист может перетащить заказ из нераспределенных в конкретную машину.
o	При дропе фронт отправляет запрос на пересчет recalculate_trip(trip_id, new_order_id).
o	Система должна ответить за < 2 сек новым порядком точек.
•	Карта: Отображение полилинии маршрута с помощью Leaflet. Цветовое кодирование: Красный — опоздание по ETA, Зеленый — запас по времени.
4.2. Мобильный чек-лист водителя (PWA)
•	Интерфейс должен работать оффлайн (Service Workers).
•	Экран «Загрузка»: Сканирование QR-кода фермера -> Появление списка товаров -> Кнопки +/- (корректировка факта).
•	Экран «Выгрузка»: Отображение ЭЦП (Touch-подпись на Canvas) -> Отправка на сервер при появлении сети.
5. Требования к стеку и окружению
•	Язык бэкенда: Python 3.11+ с FastAPI (для асинхронной работы с внешними API геокодеров).
•	База данных: PostgreSQL 15 + PostGIS.
•	Гео-сервис: Использовать API коммерческой картографии (Яндекс.Маршрутизация или аналог), так как бесплатные OSRM не дают точного ETA с учетом городских пробок.
•	Фронтенд: предложи варианты .
6. Критические сценарии и состояния (Error & Edge Cases)
1.	Срыв загрузки: Если водитель нажал «Недовес» на точке pickup, статус заказа должен смениться на dispute, а объем груза в текущем рейсе — динамически уменьшиться. Маршрут при этом не перестраивается, если это последняя точка загрузки.
2.	Конфликт окон: Если логист добавил заказ вручную в машину, но он не вписывается по времени, бэкенд должен вернуть ошибку 409 Conflict с JSON-телом, в котором указано: {"reason": "time_window_violation", "conflicting_order": "ID"}, и подсветить конфликтующую точку на карте красным.
3.	Отсутствие машин: Если сумма заказов на день превышает вместимость парка ТС, система генерирует уведомление «Требуется привлечение наемного транспорта» и переводит лишние заказы в буфер ожидания.
7. Форматы API-ответов для гео-расчетов
Для единообразной работы карты, любой ответ, содержащий координаты, должен строго соответствовать GeoJSON FeatureCollection:
{
  "type": "FeatureCollection",
  "features": [
    {
      "type": "Feature",
      "geometry": {
        "type": "Point",
        "coordinates": [37.6, 55.7]
      },
      "properties": {
        "order_id": 123,
        "type": "pickup",
        "eta": "2024-05-02T09:30:00Z",
        "status_color": "green"
      }
    }
  ]
}
